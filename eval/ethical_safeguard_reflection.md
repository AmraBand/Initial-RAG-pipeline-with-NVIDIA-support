# Ethical Safeguard Reflection

I chose a refusal-policy plus output-filtering safeguard because it is simple to implement, auditable, and effective for high-risk prompts in a small RAG pipeline. The input-stage filter blocks clearly unsafe requests before retrieval and model invocation, which reduces the chance of harmful responses and saves token usage. The output-stage filter adds defense-in-depth by catching unsafe text if it appears in generated output. This safeguard is effective for explicit harmful intent, but it has trade-offs: regex-based checks can miss nuanced harmful phrasing (false negatives) or occasionally block benign prompts containing risky keywords (false positives). A future improvement would add a lightweight safety classifier and conversation-history-aware checks to improve precision and recall.
